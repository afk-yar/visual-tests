# visual-tests — полигон визуальных задач: дизайн

Дата: 2026-06-12
Статус: принят пользователем (включая измерение «сравнение моделей»)

## Назначение

Полигон для **сравнения визуальных возможностей разных LLM-моделей**. Одной и
той же текстовой задаче («отобрази X») разные модели дают свои решения —
самодостаточные HTML-демки. Все собрано в одну витрину: слева список задач,
справа — переключатель моделей и активное решение, под ним — общий промпт под
спойлером. Цель — быстро сравнить, *как одну задачу решили разные модели*.

## Стек и режим запуска

- Чистый статичный **HTML/CSS/JS, без сборки**. Никакого npm/Node/бандлера в
  рантайме. (Node используется только как dev-time раннер unit-тестов через
  встроенный `assert` — без package.json и node_modules.)
- Работает в **двух режимах** запуска без изменений:
  - двойной клик по `index.html` (`file://`);
  - локальный сервер (`python -m http.server`, VS Code Live Server) — `http://localhost`.
- Чтобы `file://` не ломался, действуют две конвенции:
  - манифест задач — это `manifest.js` (обычный `<script>`, кладет список в
    глобальную переменную), а **не** `fetch('manifest.json')`;
  - демки **не используют** ES-module-импорты и `fetch` локальных файлов.
- Локальный сервер нужен для скриншотов/проверки демок через Playwright MCP.

Причина ограничений: при `file://` браузер блокирует программную загрузку
локальных файлов через ES-модули и `fetch`. Подключение `<script>` и показ
чужой страницы в iframe под этот запрет не попадают.

## Модель данных (ключевое)

Задача (task) содержит **общий промпт** и список **решений** (по одному на
модель). Каждое решение — самодостаточная папка-демка.

```js
window.TASKS = [
  {
    id: "double-pendulum",          // == имя папки задачи, == часть URL-хеша
    title: "Двойной маятник",
    prompt: `…единый текст промпта, который давали каждой модели…`,
    tags: ["физика", "canvas", "хаос"],
    solutions: [
      { model: "Claude Opus 4.8", slug: "opus-4.8", dir: "demos/double-pendulum/opus-4.8/" },
      // { model: "GPT-5", slug: "gpt-5", dir: "demos/double-pendulum/gpt-5/" },
    ],
  },
];
```

- `prompt` — **общий** для всех решений задачи (каждой модели давали один и тот
  же текст). Спойлер показывает именно его.
- `solutions[].dir` — папка решения; в нее всегда кладется `index.html`.
  Оболочка грузит `dir + "index.html"` (надежно и на сервере, и на `file://`).
- Список `solutions` уже готов к режиму side-by-side в будущем; пока показываем
  по одному (переключатель).

## A. Архитектура и поток данных

Единственная точка входа — `index.html` (**оболочка**, shell), master–detail:

```
┌────────────┬──────────────────────────────────────┐
│  Задачи    │ [ Opus 4.8 ][ GPT-5 ][ Gemini ] ← switch│
│  (сайдбар) │ ┌──────────────────────────────────┐   │
│ • Двойной  │ │   <iframe> с решением модели     │   │
│   маятник  │ │                                  │   │
│ • SVG конь │ └──────────────────────────────────┘   │
│            │ ▸ Промпт задачи (<details>, общий)      │
└────────────┴──────────────────────────────────────┘
```

- Сайдбар слева — список задач из `manifest.js`.
- Вверху правой части — **переключатель моделей** (сегменты/вкладки): только те
  модели, у которых есть решение для выбранной задачи. Клик меняет `iframe.src`.
- По центру — `<iframe>` с активным решением. iframe ради изоляции: свой
  JS/CSS-скоуп, демку можно открыть и отдельным файлом. Оболочка только меняет
  `iframe.src`, внутрь не читает.
- Снизу — `<details>` с общим промптом задачи (из записи задачи, не из демки).
- **URL-хеш**: `#<taskId>` (первая модель по умолчанию) или
  `#<taskId>/<modelSlug>` — дип-линк на конкретную пару задача×модель.

Ключевое свойство: iframe только **показывает**. Оболочке не нужно читать его
содержимое — поэтому все работает и через сервер, и через `file://`.

### Визуальный эталон оболочки

Внешний вид оболочки утверждён по макету `docs/superpowers/mockup-shell.html`
(сгенерирован Codex, принят пользователем как эталон). Тёмная тема, акцент —
бирюзовый (`--accent: #37c6d9`), сегментированный переключатель моделей,
рамка-«iframe» с хромом, спойлер промпта снизу. При реализации:

- инлайн-`<style>` макета переносится в `assets/shell.css` (1:1 по виду);
- статическая разметка макета превращается в управляемую данными: сайдбар и
  сегменты рендерит `shell.js` из `manifest.js`, в рамку ставится реальный
  `<iframe>`, в спойлер — текст промпта активной задачи;
- декоративная SVG-схема маятника из макета в реальной оболочке не нужна (там
  будет живой iframe), но остаётся ориентиром пропорций области просмотра.

## B. Структура папок и добавление решения

```
visual-tests/

  manifest.js                      ← window.TASKS (задачи + solutions[])
  assets/
    shell.css
    shell.js                       ← сайдбар, переключатель моделей, iframe, спойлер
    shell.test.js                  ← node-тест чистого хелпера маршрутизации
  demos/
    double-pendulum/
      opus-4.8/
        index.html                 ← самодостаточное решение
        style.css
        pendulum.js                ← чистая физика (dual-mode: браузер + node)
        pendulum.test.js           ← node-тесты физики
        app.js                     ← canvas-рендер + UI (только браузер)
  CLAUDE.md                        ← конвенции проекта
  README.md
  docs/                            ← спеки, планы, changes
```

Добавление **нового решения** существующей задачи = 2 шага:
1. положить папку в `demos/<task>/<model-slug>/` (самодостаточный `index.html`);
2. дописать запись в `solutions[]` нужной задачи в `manifest.js`.

Добавление **новой задачи** = создать `demos/<task>/…` и добавить объект задачи
в `manifest.js` (с `prompt` и хотя бы одним решением). Пересборки нет.

## C. Первая демка — двойной маятник (решение Claude Opus 4.8)

Путь: `demos/double-pendulum/opus-4.8/`. Это решение **одной модели** (моей) —
первая точка сравнения; остальные модели добавятся позже как соседние папки.

Источник постановки: myPhysicsLab (Double Pendulum), подтверждено независимыми
реализациями (skeeto/double-pendulum, cylian.org). Уравнения — каноника
лагранжевой механики; замкнутого решения нет, интегрируется численно.

### Физика

Состояние: `(θ₁, θ₂, ω₁, ω₂)`. Угловые ускорения:

```
ω₁' = [ −g(2m₁+m₂)·sinθ₁ − m₂·g·sin(θ₁−2θ₂)
        − 2·sin(θ₁−θ₂)·m₂·(ω₂²·L₂ + ω₁²·L₁·cos(θ₁−θ₂)) ]
      / [ L₁·(2m₁+m₂ − m₂·cos(2θ₁−2θ₂)) ]

ω₂' = [ 2·sin(θ₁−θ₂)·(ω₁²·L₁·(m₁+m₂) + g(m₁+m₂)·cosθ₁
        + ω₂²·L₂·m₂·cos(θ₁−θ₂)) ]
      / [ L₂·(2m₁+m₂ − m₂·cos(2θ₁−2θ₂)) ]
```

Позиции грузов (пивот в точке подвеса, y вниз):

```
x1 = L1·sinθ₁ ;  y1 = L1·cosθ₁
x2 = x1 + L2·sinθ₂ ;  y2 = y1 + L2·cosθ₂   (масштаб пикс/метр — в рендерере)
```

### Интегрирование

- **RK4** с фиксированным шагом `dt ≈ 0.005 s`.
- Шаг физики **отвязан от частоты кадров**: на кадр копим прошедшее время и
  делаем N под-шагов фиксированного dt (аккумулятор). Иначе хаотическая
  траектория «плывет» по-разному на разных мониторах.

### Рендер и UX

- `<canvas>` 2D: два стержня, два груза, затухающий след (trail) нижнего груза.
- Управление: пауза/пуск, сброс, ползунки масс `m₁,m₂` и длин `L₁,L₂`.
- Кнопка «призрак»: клон с `δθ₁ = 1e-4` — демонстрация чувствительности к
  начальным условиям (детерминированный хаос / эффект бабочки).
- Старт по умолчанию: θ₁ ≈ θ₂ ≈ 120°, ω = 0.
- Демка автономна: без npm, без module-импортов, JS — локальные `<script>`.

## Тестирование (без фреймворка)

- **Чистая физика** (`pendulum.js`) — реальный TDD через `node pendulum.test.js`
  (встроенный `assert`). Неперекрестные инварианты:
  - равновесие: вертикаль в покое → угловые ускорения = 0;
  - редукция к простому маятнику: `m₂ = 0` → `a₁ = −g·sinθ₁/L₁` (аналитика);
  - геометрия `bobPositions` (тривиально проверяемо);
  - сохранение энергии: дрейф энергии за 10 с интегрирования < 0.5 %.
- **Чистый хелпер маршрутизации оболочки** (`resolveActiveSelection`) —
  `node shell.test.js`.
- **DOM/canvas/UI** (рендер, сайдбар, переключатель, спойлер) — проверка через
  **Playwright MCP**: `python -m http.server`, навигация, скриншот, проверка
  отсутствия ошибок в консоли.

`pendulum.js` и `shell.js` — **dual-mode**: в браузере кладут API в глобал, в
node экспортируют через `module.exports`. Это дает node-тестируемость без
нарушения «без сборки» и без ES-модулей (file://-совместимость сохраняется).

## D. Проектный CLAUDE.md (содержание)

- Назначение: полигон для сравнения визуальных решений разных LLM.
- Стек: чистый статичный HTML/JS, без сборки.
- Конвенции: задача = папка `demos/<task>/` + объект в `manifest.js`; решение
  модели = подпапка `demos/<task>/<model-slug>/` + запись в `solutions[]`.
- Правило `file://`-совместимости: без ES-module-импортов и `fetch` локальных
  файлов; манифест только через `<script>`; dual-mode для тестируемых модулей.
- Как смотреть: двойной клик по `index.html` или `python -m http.server`.
- Как тестировать: `node …/*.test.js` для чистой логики, Playwright MCP для UI.

- Нет сборщика, фреймворка, роутер-библиотеки (хеш-навигация руками).
- Нет режима side-by-side в первой версии (данные к нему готовы, UI — потом).
- Нет генерации сайдбара/решений из файловой системы (реестр ведем руками).
- Нет бэкенда, БД, деплоя.
