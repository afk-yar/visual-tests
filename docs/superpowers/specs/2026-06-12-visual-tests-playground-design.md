# visual-tests — полигон визуальных задач: дизайн

Дата: 2026-06-12
Статус: принят пользователем (секции A–D)

## Назначение

Полигон для проверки визуальных возможностей модели. Модели даётся текстовая
задача («отобрази X»), её результат — самодостаточная HTML-демка. Все демки
собраны в одну витрину: слева список задач, справа активная визуализация, под
ней — исходный промпт под спойлером.

## Стек и режим запуска

- Чистый статичный **HTML/CSS/JS, без сборки**. Никакого npm/Node/бандлера.
- Работает в **двух режимах** запуска без изменений:
  - двойной клик по `index.html` (`file://`);
  - локальный сервер (`python -m http.server`, VS Code Live Server) — `http://localhost`.
- Чтобы `file://` не ломался, действуют два ограничения-конвенции:
  - манифест задач — это `manifest.js` (обычный `<script>`, кладёт список в
    глобальную переменную), а **не** `fetch('manifest.json')`;
  - демки **не используют** ES-module-импорты и `fetch` локальных файлов.
- Локальный сервер нужен только для скриншотов/проверки демок через Playwright.

Причина ограничений: при `file://` браузер блокирует программную загрузку
локальных файлов через ES-модули и `fetch` (CORS для локального источника).
Подключение `<script>` и показ чужой страницы в iframe под этот запрет не
попадают.

## A. Архитектура и поток данных

Единственная точка входа — `index.html` (**оболочка**, shell) в режиме
master–detail:

```
┌────────────┬─────────────────────────────┐
│  Задачи    │   активная визуализация      │
│  (сайдбар) │     (<iframe> с демкой)      │
│            ├─────────────────────────────┤
│            │  ▸ Промпт задачи (<details>) │
└────────────┴─────────────────────────────┘
```

- Сайдбар слева генерируется из реестра `manifest.js`.
- Справа — `<iframe>`, в который грузится самодостаточная демка. iframe выбран
  ради изоляции: у каждой демки свой JS/CSS-скоуп, ничего не конфликтует,
  демку можно открыть и отдельным файлом. Оболочка только меняет `iframe.src`.
- Под визуализацией — `<details>` с **исходным промптом**. Текст промпта живёт
  в записи реестра, а не внутри демки.
- Выбранная задача отражается в URL-хеше (`index.html#double-pendulum`) — для
  ссылки/перезагрузки на той же демке.

Ключевое свойство: iframe только **показывает**. Оболочке не нужно читать его
содержимое, поэтому всё работает и через сервер, и через `file://`.

## B. Структура папок и добавление демки

```
visual-tests/
  index.html            ← оболочка
  manifest.js           ← window.TASKS = [{ id, title, dir, prompt, tags }]
  assets/
    shell.css
    shell.js            ← рендер сайдбара, переключение iframe, заполнение спойлера
  demos/
    double-pendulum/
      index.html        ← самодостаточная демка
      pendulum.js
      style.css
  CLAUDE.md             ← конвенции проекта
  docs/                 ← спеки, планы, changes
```

Запись реестра (`manifest.js`):

```js
window.TASKS = [
  {
    id: "double-pendulum",          // == имя папки в demos/, == URL-хеш
    title: "Двойной маятник",
    dir: "demos/double-pendulum/",  // что грузится в iframe (index.html папки)
    prompt: `Отобрази в HTML5 физику двойного маятника…`, // исходный промпт
    tags: ["физика", "canvas", "хаос"],
  },
];
```

Добавление новой задачи = 2 шага:
1. положить папку в `demos/<slug>/` (самодостаточный `index.html` внутри);
2. дописать одну запись в `manifest.js`.

Пересборки нет — обновил страницу, задача появилась в сайдбаре.

## C. Первая демка — двойной маятник

Источник постановки: myPhysicsLab (Double Pendulum), подтверждено независимыми
реализациями (skeeto/double-pendulum, cylian.org). Уравнения — каноника из
лагранжевой механики; замкнутого решения нет, интегрируется численно.

### Физика

Состояние: `(θ₁, θ₂, ω₁, ω₂)`. Угловые ускорения:

```
ω₁' = [ −g(2m₁+m₂)·sinθ₁ − m₂·g·sin(θ₁−2θ₂)
        − 2·sin(θ₁−θ₂)·m₂·(ω₂²·L₂ + ω₁²·L₁·cos(θ₁−θ₂)) ]
      / [ L₁·(2m₁+m₂ − m₂·cos(2θ₁−2θ₂)) ]

ω₂' = [ 2·sin(θ₁−θ₂)·(ω₁²·L₁·(m₁+m₂) + g(m₁+m₂)·cosθ₁
        + ω₂²·L₂·m₂·cos(θ₁−θ₂)) ]
      / [ L₂·(2m₁+m₂ − m₂·cos(2θ₁−2θ₂)) ]
```

Позиции грузов (пивот в точке подвеса верхнего стержня, y вниз):

```
x1 = px + L1·s·sinθ₁ ;  y1 = py + L1·s·cosθ₁
x2 = x1 + L2·s·sinθ₂ ;  y2 = y1 + L2·s·cosθ₂   (s — масштаб пикс/метр)
```

### Интегрирование

- **RK4** с фиксированным шагом `dt ≈ 0.005 s`.
- Шаг физики **отвязан от частоты кадров**: на кадр копим прошедшее время и
  делаем N под-шагов фиксированного dt. Иначе хаотическая траектория «плывёт»
  по-разному на разных мониторах/нагрузке.

### Рендер и UX

- `<canvas>` 2D: два стержня, два груза, затухающий след (trail) нижнего груза.
- Управление: пауза/пуск, сброс, ползунки масс `m₁,m₂` и длин `L₁,L₂`.
- Кнопка «призрак»: клон с `δθ₁ = 1e-4` — наглядная демонстрация
  чувствительности к начальным условиям (эффект бабочки/детерминированный хаос).
- Старт по умолчанию из «эффектных» условий (θ₁ ≈ θ₂ ≈ 120°, ω = 0).
- Демка автономна: без npm, без module-импортов, JS — локальные `<script>`.

## D. Проектный CLAUDE.md (содержание)

- Назначение: полигон визуальных задач, генерируемых моделью.
